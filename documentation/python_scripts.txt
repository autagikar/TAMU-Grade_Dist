TAMU GRADE DISTRIBUTION — PYTHON SCRIPTS DOCUMENTATION
=======================================================

This document explains every Python file in the project in detail: what it does,
how it works, and why it was written the way it was.

The Python code covers two concerns:
  1. The SCRAPER — fetches, parses, and loads data into the database
  2. The BACKEND API — serves data from the database to the Vue frontend

================================================================================
SCRAPER
================================================================================

The TAMU grade distribution website publishes PDF reports for each combination
of year, academic term, and college. The scraper downloads those PDFs, extracts
structured grade data from them, and separately scrapes the TAMU course catalog
for course descriptions.

--------------------------------------------------------------------------------
scraper/get_college_codes.py
--------------------------------------------------------------------------------

PURPOSE:
  A one-time utility that scrapes the TAMU grade reports website to find all
  valid college codes (e.g. "EN" for Engineering, "SC" for Science). These codes
  are needed to build the download URLs.

HOW IT WORKS:
  1. Fetches the HTML of https://web-as.tamu.edu/gradereports/ using Python's
     built-in urllib.request library (no third-party HTTP library needed).
  2. Parses the HTML using Python's built-in HTMLParser class. We subclass it
     and override three methods:
       - handle_starttag: detects when we enter the college <select> dropdown
         and captures the value= attribute from each <option> tag.
       - handle_endtag: detects when the <select> closes so we stop collecting.
       - handle_data: captures the visible text inside each <option> tag.
  3. Prints every (code, name) pair it finds.

WHY THIS APPROACH:
  We could have hard-coded the college codes, but scraping them means we
  automatically pick up any new colleges TAMU adds to their system. The parsed
  list was copied into download_pdfs.py's COLLEGE_CODES constant.

--------------------------------------------------------------------------------
scraper/inspect_pdf.py
--------------------------------------------------------------------------------

PURPOSE:
  A development/debugging tool used to understand the structure of the PDFs
  before writing the full parser. It is NOT part of the production pipeline.

HOW IT WORKS:
  Opens a single PDF file using pdfplumber and prints:
  1. The raw extracted text from page 1 (to see if the PDF is text-based or
     scanned — scanned PDFs would need OCR and pdfplumber can't handle them).
  2. Any tables detected on the page (pdfplumber can detect table borders and
     return structured rows, but TAMU's PDFs don't have formal table borders so
     this returns nothing useful — we fall back to text parsing in extract.py).

KEY FINDING:
  The PDFs are text-based (not scanned), meaning pdfplumber can extract the
  text directly as a string. This meant we could parse it with regex and
  string splitting rather than needing a more complex approach.

--------------------------------------------------------------------------------
scraper/download_pdfs.py
--------------------------------------------------------------------------------

PURPOSE:
  Downloads every PDF from the TAMU grade reports website, covering all
  combinations of year (2021–2026), academic term (Spring/Summer/Fall), and
  college code.

HOW IT WORKS:
  1. Defines the list of years, terms, and college codes to try.
  2. For each combination, builds the download URL:
       https://web-as.tamu.edu/gradereports/Report?year=2026&term=SPRING&college=EN
  3. Makes an HTTP GET request using urllib.request (no third-party libraries).
  4. Validates the response before saving:
       - Must be larger than 1KB (error pages are small HTML documents)
       - Must start with the magic bytes "%PDF" (the standard PDF file header)
     This prevents saving error pages as if they were valid PDFs.
  5. Writes valid responses to raw_data/pdfs/ using the filename pattern:
       {year}_{term}_{college}.pdf   e.g. "2026_SPRING_EN.pdf"
  6. Skips files that already exist, making the script safe to re-run after
     interruptions without re-downloading everything.
  7. Waits 0.3 seconds between requests to avoid overloading the TAMU server.

NOT EVERY COMBINATION EXISTS:
  Not every year/term/college combination has data. For example, some colleges
  don't offer summer courses, or data doesn't go back as far as 2021 for all
  colleges. The validation step handles this gracefully by recording "no data"
  instead of crashing.

--------------------------------------------------------------------------------
scraper/extract.py
--------------------------------------------------------------------------------

PURPOSE:
  The core PDF parser. Reads a single PDF file and returns a list of Python
  dictionaries, one dictionary per section row. This is imported by load_to_db.py.

PDF STRUCTURE:
  Each page contains a header section and data rows.

  Header lines look like:
    COLLEGE: ENGINEERING
    DEPARTMENT: AEROSPACE ENGINEERING

  Data rows look like:
    AERO-201-500 13 20 4 0 0 37 3.215 0 0 0 0 0 37 TAKACHI TOMITA J

  The tokens (space-separated values) in each data row correspond to:
    Position 0:     Section ID (e.g. "AERO-201-500")
    Position 1:     A count
    Position 2:     B count
    Position 3:     C count
    Position 4:     D count
    Position 5:     F count
    Position 6:     A-to-F total (sum of A+B+C+D+F)
    Position 7:     GPA
    Position 8:     I (Incomplete)
    Position 9:     S (Satisfactory)
    Position 10:    U (Unsatisfactory)
    Position 11:    Q (Dropped)
    Position 12:    X (Absent from final)
    Position 13:    Total enrolled
    Position 14+:   Instructor name (raw format: "LASTNAME FIRSTNAME_INITIAL")

  Each data row is followed by a percentage row which we skip:
    35.14% 54.05% 10.81% 0.00% 0.00%

KEY FUNCTIONS:

  format_instructor(raw)
    Converts the raw instructor name format from the PDF to a readable format.
    The PDF stores names as "LASTNAME FIRST_INITIAL" (e.g. "TAKACHI TOMITA J").
    This function converts it to "J. Takachi Tomita" by treating the last token
    as the first initial and title-casing everything.

  parse_section_id(section_id)
    Splits "AERO-201-500" into ("AERO-201", "500") — course code and section number.
    This is done by splitting on dashes and treating the last part as the section.

  is_data_row(line)
    Uses a regular expression to detect if a line is a data row. The pattern
    checks that the line starts with an uppercase department code, a dash, digits,
    and then a space followed by a number. This distinguishes data rows from
    header lines, blank lines, and page numbers.

  is_percentage_row(line)
    Detects percentage rows (which start with a number followed by "%" like
    "35.14%"). These are skipped because we already have the raw counts.

  parse_data_row(line, college, department, semester)
    Splits the line into tokens and maps each position to its field name.
    GPA validation: if the parsed GPA value is outside 0–4.0, it's stored as
    None (NULL in the database). This handles rare malformed rows where the
    tokens are misaligned and would otherwise write garbage GPA values.

  semester_from_filename(filename)
    Derives the semester string from the filename convention.
    "2026_SPRING_EN.pdf" → "Spring 2026"

  extract(pdf_path)
    The main function that ties everything together:
    1. Opens the PDF using pdfplumber.
    2. Loops over every page, extracts its text.
    3. Tracks the current college and department from header lines.
    4. For each data row, calls parse_data_row() and appends the result.
    5. Returns the list of all parsed section dictionaries.

--------------------------------------------------------------------------------
scraper/load_to_db.py
--------------------------------------------------------------------------------

PURPOSE:
  Reads every PDF in the raw_data/pdfs/ directory, parses them using extract.py,
  and bulk-inserts all records into the PostgreSQL database.

HOW IT WORKS:
  1. Opens a single database connection (reusing it for all files is much more
     efficient than opening a new connection per PDF).
  2. Iterates over all PDFs in the directory in sorted order.
  3. For each PDF, calls extract() to get a list of section dicts.
  4. Uses psycopg2's executemany() to insert the entire batch in one call.
  5. Calls conn.commit() after each PDF so that progress is saved even if the
     script crashes partway through.
  6. Prints progress as each file is processed.

DATABASE CONNECTION:
  If the DATABASE_URL environment variable is set (e.g. pointing to Supabase),
  it connects using that URL directly. Otherwise falls back to the local Docker
  config (localhost:5432, user "tamu", password "devpassword", db "grades").

NOTE ON S/U GRADED COURSES:
  Some TAMU courses use Satisfactory/Unsatisfactory grading instead of letter
  grades. For these sections, the PDF records gpa=0 since no numeric GPA is
  assigned. The application detects gpa=0 as S/U and excludes these sections
  from all GPA calculations and rankings to prevent them from artificially
  dragging down averages.

--------------------------------------------------------------------------------
scraper/fetch_descriptions.py
--------------------------------------------------------------------------------

PURPOSE:
  Scrapes course title, credits, lecture hours, lab hours, description, and
  prerequisites from the TAMU undergraduate and graduate course catalogs and
  loads them into the course_descriptions lookup table. This script is separate
  from the PDF scraper because it targets a different data source (the web
  catalog rather than PDF reports).

WHY A SEPARATE LOOKUP TABLE:
  The sections table has ~99,000 rows, each representing one section of one
  course in one semester. Storing description data in sections would mean
  duplicating it thousands of times. A separate course_descriptions table with
  one row per course code is cleaner and more efficient. The two tables are
  linked logically at query time using the course code string — no foreign key
  constraint is needed.

HOW IT WORKS:
  1. Queries the sections table to find all distinct course prefixes
     (e.g. "BMEN", "CSCE") using SPLIT_PART on the course code.
  2. For each prefix, hits both the undergraduate and graduate catalog URLs:
       https://catalog.tamu.edu/undergraduate/course-descriptions/{prefix}/
       https://catalog.tamu.edu/graduate/course-descriptions/{prefix}/
  3. Parses each page with BeautifulSoup, finding all div.courseblock elements.
  4. For each course block, extracts:
       - Course code: from the title text ("BMEN 291" → stored as "BMEN-291")
       - Title: the course name portion of the title line
       - Credits: parsed from "Credits 3" or "3 Credits" patterns in the description
       - Lecture hours: parsed from "N Lecture Hours" pattern
       - Lab hours: parsed from "N Lab Hours" pattern
       - Description: the body text after stripping the credits/hours header
       - Prerequisites: extracted from the "Prerequisite: ..." sentence at the end
  5. Upserts each row using ON CONFLICT (course_code) DO UPDATE so the script is
     safe to re-run — existing rows are updated, not duplicated.
  6. Waits 0.3 seconds between page requests to be polite to the TAMU server.

KEY PARSING DETAIL:
  The TAMU catalog places credits, hours, description, and prerequisites all in
  a single .courseblockdesc paragraph rather than in separate HTML elements.
  The parsing extracts each field using regex, strips the credits/hours header
  from the start of the text, and strips the prerequisite sentence from the end
  to isolate the clean description body.

UPSERT SQL:
  Uses INSERT ... ON CONFLICT (course_code) DO UPDATE SET ... so that re-running
  the script refreshes data without creating duplicates. This is important because
  the catalog updates each semester.

DATABASE CONNECTION:
  Uses the same DATABASE_URL environment variable pattern as load_to_db.py.
  Run with DATABASE_URL set to target Supabase, or without it to target the
  local Docker database.

================================================================================
BACKEND API
================================================================================

The backend is a FastAPI application running inside Docker locally and on
Railway in production. It connects to PostgreSQL (local Docker or Supabase)
and exposes a REST API that the Vue frontend consumes.

--------------------------------------------------------------------------------
backend/app/database.py
--------------------------------------------------------------------------------

PURPOSE:
  Sets up the SQLAlchemy database connection that all API endpoints share.

KEY CONCEPTS:

  DATABASE_URL environment variable:
    The URL is read from the environment, not hard-coded. In docker-compose.yml,
    it's set to "postgresql://tamu:devpassword@db:5432/grades". The hostname "db"
    works because Docker's internal network resolves container names to their
    IP addresses automatically. In production, Railway injects the Supabase URL.

  engine:
    SQLAlchemy's engine manages the actual database connection pool.

  SessionLocal:
    A factory that creates new database sessions. Each HTTP request gets its
    own session via the get_db() dependency.

  get_db():
    A FastAPI dependency function injected into route handlers via Depends(get_db).
    The try/finally pattern guarantees the session is always closed after the
    request, even if the handler raises an exception. This prevents connection leaks.

--------------------------------------------------------------------------------
backend/app/models.py
--------------------------------------------------------------------------------

PURPOSE:
  Defines the SQLAlchemy ORM models that map to the PostgreSQL tables.

SECTION MODEL:
  Maps to the "sections" table. Each attribute corresponds to a column.

  Column notes:
    - gpa is Numeric (not Float) to avoid floating-point precision issues.
    - a_to_f is the sum of A+B+C+D+F grades stored separately because TAMU
      publishes it directly and it's used as the weight in GPA calculations.
    - s/u columns track Satisfactory/Unsatisfactory outcomes. When a section is
      S/U graded, gpa=0 and a_to_f=0, while s and u hold the student counts.

COURSEDESCRIPTION MODEL:
  Maps to the "course_descriptions" lookup table populated by fetch_descriptions.py.

  Columns:
    course_code   TEXT PRIMARY KEY  — e.g. "BMEN-291"
    title         TEXT              — e.g. "Biomedical Engineering Laboratory I"
    credits       NUMERIC(4,1)      — e.g. 3.0 (NUMERIC not INTEGER to handle 1.5-credit courses)
    lecture_hours SMALLINT          — e.g. 3
    lab_hours     SMALLINT          — e.g. 1
    description   TEXT              — catalog course description body
    prerequisites TEXT              — prerequisite text, parsed from catalog

--------------------------------------------------------------------------------
backend/app/schemas.py
--------------------------------------------------------------------------------

PURPOSE:
  Defines the SectionResponse Pydantic model which controls what data the API
  returns for section rows.

HOW IT WORKS:
  FastAPI uses Pydantic for output serialization. When a route returns
  list[SectionResponse], FastAPI reads SQLAlchemy model attributes, validates
  types, and serializes to JSON automatically.

  from_attributes = True (formerly orm_mode) tells Pydantic to read from
  SQLAlchemy model attributes rather than expecting plain dictionaries.

  Optional fields (gpa, instructor) handle NULL database values by allowing
  them to become null in the JSON response.

--------------------------------------------------------------------------------
backend/app/main.py
--------------------------------------------------------------------------------

PURPOSE:
  The FastAPI application entry point. Sets up CORS middleware and registers
  the API router.

CORS MIDDLEWARE:
  The Vue frontend and API run on different origins (different ports in dev,
  different domains in production). Without CORS middleware, browsers block
  all cross-origin requests. The middleware is configured to allow requests
  from both localhost:5173 (local dev) and the deployed Vercel frontend URL.

--------------------------------------------------------------------------------
backend/app/routers/sections.py
--------------------------------------------------------------------------------

PURPOSE:
  All API route handlers. Every endpoint the Vue frontend calls is defined here.

ENDPOINTS:

  GET /api/colleges
    Returns all distinct college names. Used for verifying data loaded correctly.
    Not consumed by the frontend UI.

  GET /api/departments?college=ENGINEERING
    Returns distinct department names, optionally filtered by college.

  GET /api/semesters
    Returns all distinct semester strings (e.g. "Spring 2026").
    The frontend sorts these by a custom chronological key after receiving them.

  GET /api/courses/search?q=csce
    Autocomplete for course codes. Normalizes the query (uppercase, spaces→dashes)
    so "csce 121" and "CSCE-121" both match. Requires q ≥ 2 characters.

  GET /api/courses/description?course=CSCE-120
    Looks up catalog description data for a single course from the
    course_descriptions table. Returns null if the course has no catalog entry.
    Normalizes the course code the same way as the search endpoint.

  GET /api/instructors/search?q=smith
    Autocomplete for instructor names. Case-insensitive substring search.
    Returns only non-empty values.

  GET /api/sections?course=CSCE-121&semester=Spring+2026&instructor=J.+Smith
    The main data endpoint. Returns all matching section rows.
    At least one of course or instructor is required to prevent returning all
    99k rows. Course codes are normalized on input.

  GET /api/rankings?department=COMPUTER+SCIENCE+%26+ENGINEERING
    Returns one row per professor in the department with aggregated stats:
    avg_gpa, total_students, sections_count, a_percent, and a computed score.

    PROFESSOR SCORE FORMULA:
      score = round((avg_gpa / 4.0) * 70 + a_percent * 30)
      - 70% weight on average GPA (scaled to 0–100)
      - 30% weight on percentage of students who received A grades
      - Capped at 100
      - S/U sections (gpa=0) are excluded from the calculation via
        Section.gpa > 0 filter so they don't drag down professor scores.

    Only professors with ≥ 2 sections are included (filters out one-off visitors).
    Default sort: highest score first.

  GET /api/course-rankings?department=COMPUTER+SCIENCE+%26+ENGINEERING
    Returns one row per course in the department with aggregated stats:
    avg_gpa, total_students, sections_count, a_percent, f_percent, q_percent.

    S/U sections (gpa=0) are excluded via Section.gpa > 0 filter so S/U-only
    courses don't appear as the "hardest" courses in the rankings.
    Default sort: lowest avg_gpa first (hardest courses at the top).
